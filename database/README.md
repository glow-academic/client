# Database Management

This directory contains all database-related operations for the Glow project. The database is the single source of truth - schema and seed data are exported from the live database.

## Database-First Approach

The database is the source of truth. Schema definitions and seed data are generated from the live database using export scripts, not maintained separately.

### Exporting Database Components

Use `make export-db` to export different parts of the database:

```bash
make export-db schema       # Export schema.sql (DDL only)
make export-db base         # Export base.sql (models, agents, default prompts)
make export-db university   # Export university.sql (practice simulations, scenarios, etc.)
make export-db organization # Export organization.sql (currently no-op)
```

These commands generate SQL files in the `database/` directory that can be used to recreate the database structure and seed data.

## Workflow

### 1. Default Start (Latest Backup)
```bash
npm run start
# or
yarn start
# or
make restore-db
```
- Finds the latest backup from `history/` folder
- Restores the backup to a fresh database
- Keeps running until interrupted (creates backup on exit)

### 2. Fresh Start (Interactive Setup)
```bash
make fresh-db
```

This runs an interactive setup that:
1. Prompts for configuration (NODE_ENV, PORT, ORIGIN, APP_PREFIX)
2. Asks for institution type (organization or university)
3. Configures color scheme (presets or custom)
4. Sets up auth providers (Google, Microsoft) with encrypted credentials
5. Sets up AI providers (OpenAI, Gemini) with encrypted API keys
6. Generates a timestamped combined seed file in `database/seeds/seed_YYYYMMDD_HHMMSS.sql`
7. Updates `.env` file with all configuration

The generated seed file contains:
- Schema (from schema.sql if it exists)
- Base seed data (from base.sql if it exists)
- Institution seed data (from university.sql or organization.sql if it exists)
- Generated INSERT statements for settings, auth providers, and AI providers

To load the generated seed file:
```bash
cd database && yarn start:clean
```

### 3. Migration Mode (Apply Manual Migrations)
```bash
npm run migrate
# or
yarn migrate
# or
make migrate-db
```
- Applies migration files from `migrate/` folder to the existing database
- Migration files are applied in alphabetical order

### 4. Connect to Database
```bash
npm run connect
# or
yarn connect
# or
make connect-db
```
- Opens an interactive psql session to the existing database

## Directory Structure

```
database/
├── scripts/
│   ├── export-db.sh          # Consolidated export script (schema, base, university, organization)
│   ├── setup-fresh-db.sh     # Interactive setup script for fresh database

│   ├── start.sh               # Database management script
│   └── ...
├── seeds/                     # Timestamped seed files (generated by fresh-db)
│   ├── seed_20250115_143022.sql
│   ├── seed_20250115_150530.sql
│   └── ...
├── migrate/                   # Manual migration SQL files
├── history/                    # Database backups (auto-created)
└── package.json               # Database-specific dependencies
```

## Export Workflow

1. **Make changes to database**: Use migrations or direct SQL
2. **Export schema**: `make export-db schema` to generate schema.sql
3. **Export seed data**: `make export-db base` and `make export-db university` (or organization)
4. **Use exports**: The exported files can be used to recreate the database structure

## Fresh Database Setup Workflow

1. **Export current database** (if you have one):
   ```bash
   make export-db schema
   make export-db base
   make export-db university  # or organization
   ```

2. **Run interactive setup**:
   ```bash
   make fresh-db
   ```
   This generates a timestamped seed file with all your configuration.

3. **Load seed file**:
   ```bash
   cd database && yarn start:clean
   ```
   This loads the latest seed file from `database/seeds/`.

## Seed File History

Each `make fresh-db` run creates a new timestamped seed file:
- Format: `seed_YYYYMMDD_HHMMSS.sql` (e.g., `seed_20250115_143022.sql`)
- Files stored in `database/seeds/` directory
- Allows tracking of what was seeded at different points in time
- Old seed files can be referenced for debugging or rollback scenarios

## Migration Workflow

1. **Create Migration File**: Create a new SQL file in `migrate/` folder (e.g., `migrate/1_add_new_table.sql`)
2. **Write Migration SQL**: Use `DO $$ BEGIN ... END $$` blocks for conditional DDL
3. **Apply Migration**: Run `yarn migrate` or `make migrate-db` to apply migrations
4. **Export Updated Schema**: After migration, run `make export-db schema` to update schema.sql
5. **Restart Services**: Restart services to pick up schema changes

## Backup System

- Backups are automatically created in `history/` folder with timestamp
- Backups are created:
  - Before any database operation (in clean mode)
  - When the database process exits (in clean mode)
- Latest backup is used when starting without `--clean` flag

## Environment Variables

Required environment variables:
- `DB_USER` - Database user (default: myuser)
- `DB_PASSWORD` - Database password (default: mypassword)
- `DB_NAME` - Database name (default: mydb)
- `DB_HOST` - Database host (default: localhost)
- `DB_PORT` - Database port (default: 5432)
- `SECRET_KEY` - Required for encrypting API keys and secrets (used by encrypt-keys.js)

## Notes

- All migrations are manual SQL files in `migrate/` folder
- Migration files are applied in alphabetical order
- The system preserves data through backups during migrations
- Server uses asyncpg for all database operations
- Database is the source of truth - schema.sql and seed files are generated from the live database
- Seed files are timestamped and stored in `database/seeds/` for history tracking
